## Codebase Patterns
- Structured logging: `backend/src/lib/log.ts` exposes `log.info/warn/error` emitting single-line JSON to stdout/stderr. `requestId` is carried via AsyncLocalStorage (set by `middleware/requestId.ts`); calls outside request scope (cron/scheduler/worker) omit it automatically. Morgan is configured in `index.ts` with a JSON-shaped format string and a custom `:requestId` token; mount order is `requestId` → `morgan` so the token resolves.
- Mongoose collection naming: model names are pluralised + lowercased. `ForecastSnapshot` → `forecastsnapshots`. When inspecting indexes via mongosh, query the lowercase plural form.
- Docker dev workflow: backend/UI containers each have a named volume for `node_modules`. After `npm install` on the host, **also run `docker exec surfsapp-backend-1 npm install`** (or restart the container if the volume is fresh) — host-installed deps don't appear inside the container.
- ESLint flat config in both packages (ui/eslint.config.js, backend/eslint.config.mjs). Backend uses .mjs because backend/package.json is CommonJS; UI uses .js because ui/package.json has "type": "module".
- Single root .prettierrc is the source of truth for formatting; per-package .prettierignore files exist because Prettier doesn't traverse parents for ignore files (only for config).
- Vue files: `no-undef` is disabled in eslint config because TypeScript handles type-name resolution (e.g. `GeoJSON` namespace types from maplibre-gl).
- `backend/src/db/mongo-init.js` is a MongoDB shell script (not Node) — exclude from ESLint.
- Lint commands: `cd ui && yarn lint`, `cd backend && npm run lint`. Format: same with `format` instead of `lint`.
- TS strictness: both tsconfigs enable `strict`, `noUncheckedIndexedAccess`, `noFallthroughCasesInSwitch`. Indexed access (`arr[i]`, `record[key]`) returns `T | undefined` — guard with `?? fallback`, narrow via destructuring (`const first = arr[0]; if (!first) return;`), or hoist to a local before use. Avoid `!` non-null assertions; prefer explicit fallbacks.
- Root package.json (private, version 0.2.0) hosts husky + lint-staged + tsc-files + a pinned `typescript@~5.9.3` so pre-commit type checks use the same TS version as the workspaces (tsc-files resolves typescript from its nearest node_modules — without a root pin, it pulls TS 6.x from npm latest and the strict-deprecations error TS5107 fires on `moduleResolution: "node"`).
- Pre-commit pipeline (`.husky/pre-commit` → `npx lint-staged`) is configured in root `package.json` `lint-staged` field. Globs are directory-scoped (`ui/**/*.{ts,tsx}`, `ui/**/*.vue`, `backend/**/*.ts`, `*.{js,json,md}`) because eslint flat config and tsc-files both resolve config relative to cwd — so the commands `cd` into the package dir via `bash -c '...'` wrappers.
- For `.vue` files the pre-commit runs `yarn typecheck` (full project vue-tsc) rather than per-file vue-tsc; vue-tsc has no stable per-file mode that preserves project tsconfig.

  - `backend/src/services/scoringEngine.ts`: in `computeDailySummaries`, hoisted `hours[0]` to `first`, guarded the inner-window loop's `hours[j]`, and used `hours[bestStart] ?? first` / `hours[bestEnd] ?? first` for the summary timestamps.

- **Learnings for future iterations:**
  - `noUncheckedIndexedAccess` is the bigger of the two flags by far — `noFallthroughCasesInSwitch` produced zero errors in this codebase. Plan future work assuming most fixes will cluster around array/record access sites.
  - Vue `computed` blocks lose narrowing across boundaries: `hasForecast` (computed) gating `swellText` (also computed) does NOT propagate `r.swell_wave_height` being defined. Use `?? fallback` inside each computed instead of relying on a sibling guard.
  - When fixing `Record<string, T>` access errors, check whether the underlying type should actually be `Partial<Record<…, T>>` — `noUncheckedIndexedAccess` essentially makes record access behave that way at the call site, so callers must already handle undefined.

- **Learnings for future iterations:**
  - **TypeScript version pin at root is non-obvious but mandatory.** tsc-files resolves `typescript` from its own node_modules dir (root), not from the package whose tsconfig it's loading. Without a root pin, npm pulled TS 6.0.3 (latest), which fires hard error TS5107 on `moduleResolution: "node"` even though TS 5.9 only warns. Symptom: `tsconfig.<rand>.json(5,25): error TS5107` masking real type errors. Fix: pin `typescript` at root to match the workspaces' version.
  - **lint-staged static config in package.json forces shell wrappers.** Functions are only available in `lint-staged.config.{js,mjs,cjs}`. To `cd` into ui/ or backend/ before invoking eslint/tsc-files, wrap each command in `bash -c 'cd <pkg> && <tool> "$@"' --` — the trailing `--` becomes `$0` so user file paths land in `"$@"`.
  - **ESLint flat config is cwd-bound, not file-bound.** `eslint <file>` from a directory without an `eslint.config.{js,mjs}` does NOT walk up to find one. lint-staged wrappers must `cd` into the package containing the config.
  - **Prettier `.prettierignore` is also cwd-bound** — root .prettierignore is found from root cwd, but per-package ones aren't. The pre-commit runs prettier from root cwd, so root .prettierignore applies (acceptable: staged files are unlikely to match the per-package ignores).
  - **vue-tsc has no clean per-file mode.** `vue-tsc <file>` drops tsconfig context and breaks import resolution. Running `yarn typecheck` (full project) on .vue stages is the pragmatic choice; project is small enough to stay sub-2s warm.
  - **Husky v9 setup is `prepare: "husky"` (no args).** No more `husky install`. The hook directory is `.husky/_` (managed by husky, gitignored via husky's own .gitignore) and user hooks live in `.husky/pre-commit` (committed).
  - **tsc-files leaks tmp tsconfigs on Ctrl-C** — it cleans up on normal exit but a kill -9 leaves `tsconfig.<random>.json` files in the package dir. If you see these, add a glob to .gitignore.

- **Learnings for future iterations:**
  - **Express type augmentation needs `declare module 'express-serve-static-core'`, not `'express'`.** `express`'s public types re-export `Request` from `express-serve-static-core`, so augmenting `'express'` doesn't extend `req.requestId` in router handlers. Augment the core module.
  - **AsyncLocalStorage works through morgan's `res.on('finish')` automatically.** Node maintains the ALS context across event-emitter listeners attached within the context. So `log.*` called from morgan tokens (or anywhere downstream of the `runWithRequestContext` call) sees the requestId without explicit propagation.
  - **Morgan tokens emit raw text into the format string.** To produce valid JSON, write the format with the field types in mind: string fields as `"key":":token"`, numeric fields as `"key"::token` (no quotes). `:status` and `:response-time` are numeric; `:requestId`, `:method`, `:url` are strings.
  - **Mongoose pluralises model names to lowercase for the collection.** `mongoose.model('ForecastSnapshot', …)` → `forecastsnapshots`, NOT `forecastSnapshots`. Trips up mongosh queries when verifying indexes.
  - **Docker bind-mounted node_modules are container-local.** `npm install` on the host went into `./backend/node_modules`, but the backend container has a named volume (`backend-node-modules`) mounted there, hiding the host install. Either `docker exec surfsapp-backend-1 npm install`, or rebuild the container so the Dockerfile reinstalls. Neither restart alone nor `docker compose up` picks up host-side `npm install`.
  - **morgan's `:url` token is the path + query, not just the path.** Acceptable for our format string; `path` is a slight misnomer if requests carry queries. Rename to `url` later if it matters.
  - **Future story US-005 (worker offload) reuses the log wrapper as required by AC.** The wrapper already handles non-request scope (no requestId field emitted), so worker code can `import { log }` and call `log.info` without ceremony.
