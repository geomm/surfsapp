## Codebase Patterns
- Vite v5 `build.chunkSizeWarningLimit` compares against **raw** chunk size (`chunk.code.length / 1000` in `node_modules/vite/dist/node/chunks/dep-*.js`), NOT gzip. PRD specs that quote a gzip-derived budget value need to be applied against raw size, or the warning fires unconditionally on the maplibre-gl chunk (1,047 kB raw / 283 kB gzip).
- Structured logging: `backend/src/lib/log.ts` exposes `log.info/warn/error` emitting single-line JSON to stdout/stderr. `requestId` is carried via AsyncLocalStorage (set by `middleware/requestId.ts`); calls outside request scope (cron/scheduler/worker) omit it automatically. Morgan is configured in `index.ts` with a JSON-shaped format string and a custom `:requestId` token; mount order is `requestId` → `morgan` so the token resolves.
- Mongoose collection naming: model names are pluralised + lowercased. `ForecastSnapshot` → `forecastsnapshots`. When inspecting indexes via mongosh, query the lowercase plural form.
- Docker dev workflow: backend/UI containers each have a named volume for `node_modules`. After `npm install` on the host, **also run `docker exec surfsapp-backend-1 npm install`** (or restart the container if the volume is fresh) — host-installed deps don't appear inside the container.
- ESLint flat config in both packages (ui/eslint.config.js, backend/eslint.config.mjs). Backend uses .mjs because backend/package.json is CommonJS; UI uses .js because ui/package.json has "type": "module".
- Single root .prettierrc is the source of truth for formatting; per-package .prettierignore files exist because Prettier doesn't traverse parents for ignore files (only for config).
- Vue files: `no-undef` is disabled in eslint config because TypeScript handles type-name resolution (e.g. `GeoJSON` namespace types from maplibre-gl).
- `backend/src/db/mongo-init.js` is a MongoDB shell script (not Node) — exclude from ESLint.
- Lint commands: `cd ui && yarn lint`, `cd backend && npm run lint`. Format: same with `format` instead of `lint`.
- TS strictness: both tsconfigs enable `strict`, `noUncheckedIndexedAccess`, `noFallthroughCasesInSwitch`. Indexed access (`arr[i]`, `record[key]`) returns `T | undefined` — guard with `?? fallback`, narrow via destructuring (`const first = arr[0]; if (!first) return;`), or hoist to a local before use. Avoid `!` non-null assertions; prefer explicit fallbacks.
- Root package.json (private, version 0.2.0) hosts husky + lint-staged + tsc-files + a pinned `typescript@~5.9.3` so pre-commit type checks use the same TS version as the workspaces (tsc-files resolves typescript from its nearest node_modules — without a root pin, it pulls TS 6.x from npm latest and the strict-deprecations error TS5107 fires on `moduleResolution: "node"`).
- Pre-commit pipeline (`.husky/pre-commit` → `npx lint-staged`) is configured in root `package.json` `lint-staged` field. Globs are directory-scoped (`ui/**/*.{ts,tsx}`, `ui/**/*.vue`, `backend/**/*.ts`, `*.{js,json,md}`) because eslint flat config and tsc-files both resolve config relative to cwd — so the commands `cd` into the package dir via `bash -c '...'` wrappers.
- For `.vue` files the pre-commit runs `yarn typecheck` (full project vue-tsc) rather than per-file vue-tsc; vue-tsc has no stable per-file mode that preserves project tsconfig.
- Worker_threads + tsx coexistence: `backend/src/services/forecastWorker.ts` resolves the worker path via `__filename.endsWith('.ts')` (tsx preserves the original extension at runtime). In dev it spawns the worker with `execArgv: ['-r', 'tsx/cjs']` so the worker can `import` TS files without a build step; in compiled output (`dist/workers/forecastFetcher.worker.js`) `execArgv` is empty. Each worker thread has its own JS context, so it must call `connectDB()` itself — Mongoose state does NOT cross thread boundaries.
- Forecast fetch trigger pattern: callers (cron in `scheduler.ts`, admin route in `routes/admin.ts`) go through `triggerForecastFetch()` from `services/forecastWorker.ts`. The wrapper coalesces overlapping triggers via an `inflight` promise (only one fetch cycle runs at a time) and tracks per-job promises in a Map keyed by `jobId` for clean message routing.

Lazily calls `connectDB()` on first trigger (each worker has its own Mongoose connection — state does not cross thread boundaries), then runs `fetchAllBeaches()` on every `{type:'trigger', jobId}` message. Posts back `{type:'done'|'error', jobId, error?}`. Logs `forecast_fetch_worker_start`/`forecast_fetch_worker_finish` via `log.info`; the existing per-beach logs inside `fetchAllBeaches` already use `log.info` so AC's "use the log wrapper from US-004 inside the worker" is satisfied.

Resolves the worker path through `__filename.endsWith('.ts')` so `tsx --watch src/index.ts` (dev) and `node dist/index.js` (post-build) both work — dev passes `execArgv: ['-r', 'tsx/cjs']`, prod passes none. Coalesces overlapping triggers behind a single `inflight` promise.
- Rewired `backend/src/scheduler.ts` and `backend/src/routes/admin.ts` to call `triggerForecastFetch()`. Both the cron entry point and `POST /admin/fetch-forecasts` go through the same code path; neither runs the fetcher inline on the HTTP main thread anymore. Admin route's catch is a no-op (worker logs its own errors; the HTTP caller already received a 200 fire-and-forget response).

  - `bench-during-fetch.mjs` — concurrently spams `POST /admin/fetch-forecasts` (`BENCH_TRIGGERS` controls fan-out; loop wakes every 50ms) and runs the same sample loop against `/beaches` and `/beaches/:id/forecast`.
  - `bench-single-fetch.mjs` — fires one trigger then samples (closer to real cron load).

  - Idle:                     /beaches p95=9.4   /beaches/:id p95=2.8
  - During fetch (inline):    /beaches p95=27.2  /beaches/:id p95=15.7
  - During fetch (worker):    /beaches p95=10.7  /beaches/:id p95=3.7
  - Improvements vs inline: ~60% on /beaches, ~76% on /beaches/:id. After-offload p95 deltas vs idle (1.3ms, 0.9ms) are within bench noise floor (idle p95 itself varied ±2ms run-to-run).

- Files added: `backend/src/workers/forecastFetcher.worker.ts`, `backend/src/services/forecastWorker.ts`, `scripts/bench-backend.mjs`, `scripts/bench-during-fetch.mjs`, `scripts/bench-single-fetch.mjs`.

- Quality gates: `cd backend && npm run typecheck && npm run lint` ✅. (Run inside `surfsapp-backend-1` because the named volume holds the only authoritative `node_modules`.)
- **Learnings for future iterations:**
  - **tsx + worker_threads needs `__filename` extension detection.** tsx preserves `.ts` on `__filename` at runtime, so `__filename.endsWith('.ts')` reliably distinguishes dev (tsx) from prod (compiled). Pass `execArgv: ['-r', 'tsx/cjs']` only when the worker file is a `.ts` source — otherwise the spawned worker tries to load the JS Standard Library through tsx and ignores `dist/`.
  - **Mongoose state does not cross thread boundaries.** Each `worker_threads.Worker` has its own JS context, so `mongoose.connection` in the worker is separate from the main thread's. The worker must call `connectDB()` itself. The two connections share only the underlying mongod (different sockets, different Mongoose models).
  - **Outbound HTTP from a worker doesn't block main-thread sockets.** libuv is per-process but TCP I/O is non-blocking, so the worker's 16+ outbound `fetch()`s to open-meteo don't slow `/beaches` reads. The remaining post-offload p95 delta on `/beaches` is mongod-side contention from the worker's writes interleaving with the main thread's reads — not solvable at the Node level.
  - **Bench harness load matters at sub-10ms latencies.** The `BENCH_TRIGGERS=5` mode (5 concurrent admin POSTs every 50ms) doubled p95 on `/beaches` because morgan + JSON middleware on the main thread were running at ~100 POST/s. Single-trigger mode (`BENCH_TRIGGERS=1`) keeps the worker constantly fetching while leaving the main thread free, which matches a real cron cycle better.
  - **For sustained-load benchmarks, coalesce duplicate triggers.** `triggerForecastFetch()` returns the same `inflight` promise for overlapping calls, so even at 20 POST/s the worker runs back-to-back without stacking. Without coalescing, a clogged trigger queue would defeat the offload entirely.
  - **AsyncLocalStorage requestId context does NOT cross worker_threads boundaries.** Worker logs (`forecast_fetch_worker_start`, per-beach logs) emit without `requestId` because the `runWithRequestContext` set in main-thread middleware doesn't propagate across `worker.postMessage`. This matches the `log.ts` design (request-scope id only) and is documented in US-004 learnings.
  - **Worker error handling needs both `'error'` and `'exit'` listeners.** A `'error'` event fires for sync throws inside the worker; an unclean `'exit'` (non-zero code) can also happen without a prior `'error'`. Both reject any pending jobs and clear `workerInstance` so the next trigger respawns.

## 2026-05-02 - US-007 (Bundle-size baseline and perf budget in vite.config.ts)
- Captured fresh `yarn build` output (commit `3881cf3` plus US-007 changes). Largest chunk: `maplibre-gl-<hash>.js` at 1,047.86 kB raw / 283.21 kB gzip. Entry chunk drifted slightly from the US-006 docs snapshot (now 263.91 kB raw / 91.66 kB gzip vs the earlier 458/159) — included a fresh per-chunk markdown table in `docs/perf-baseline.md` under a new "Bundle size budget (US-007)" subsection.
- Set `build.chunkSizeWarningLimit: 1155` in `ui/vite.config.ts` with the AC-required comment. Computation: 1,047.86 → 1,050 (round up to nearest 10) → ×1.10 = 1,155 kB.
- **Deviation from AC, documented in perf-baseline.md and the commit message.** AC instructs computing the budget from the largest chunk's gzip size (283 → 290 → 319 kB). vite v5's `chunkSizeWarningLimit` is compared against raw bytes (`chunk.code.length / 1000`, see `node_modules/vite/dist/node/chunks/dep-BK3b2jBa.js:17973-18001`), so a 319 kB raw threshold would warn on every build (maplibre-gl raw is 1,047 kB) and contradict the "yarn build completes without any chunk-size warnings" AC. Applied the same round-up + 10% formula to raw size instead. Headroom on gzip is comparable: 1,155 kB raw ≈ 312 kB gzip (~10% over current 283).
- Manual verification: `yarn build` post-change → no `(!) Some chunks are larger than …` line; build completes cleanly in ~2s.
- Files modified: `ui/vite.config.ts`, `docs/perf-baseline.md`, `prd.json`, `progress.txt`.
- Quality gates: `cd ui && yarn build && yarn typecheck && yarn lint` ✅.
- **Learnings for future iterations:**
  - **Vite v5 chunkSizeWarningLimit is raw-size, not gzip.** Confirmed in `node_modules/vite/dist/node/chunks/dep-*.js` — `entry.size` is `chunk.code.length` and the warning fires when `entry.size / 1000 > chunkLimit`. Anyone copy-pasting a "set the budget to gzip + 10%" recipe from a blog post will get a budget that always trips when bundling a dependency like maplibre-gl. Either compute from raw or accept the warning.
  - **PRD numbers in baseline docs go stale fast.** The US-006 chunk table claimed entry was 458 kB raw / 159 kB gzip; six commits later (with no obvious source-side cause) it's 263 / 92. Don't reuse US-006's numbers for US-007 — re-run `yarn build` and re-record. Likely cause is dependency-graph reshuffling after the maplibre CSS move.
  - **The `build` key goes inside `defineConfig({...})` at the top level, not inside `plugins`.** Easy to land it in the wrong scope and silently no-op.
