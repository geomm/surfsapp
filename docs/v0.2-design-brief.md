# surfsapp — v0.2 Plan (Iteration 1 Improvements)

**Goal:** Harden the foundations laid in MVP/Phase 13 and turn surfsapp from a working prototype into a tool surfers can trust day-to-day. The focus is on (1) code/architecture health, (2) scoring quality you can validate against reality, (3) richer forecast visualisations to support "wave hunting", and (4) a more focused, exploration-friendly map.

**Non-goals for v0.2:** user accounts, push notifications, multi-region expansion, Capacitor native packaging — all deferred to v0.3+.

## Theme 1 — Code health & performance (P0)

Foundation work. Pays back across every other theme.

### US-001 (P0) — Dev experience baseline
**As a contributor, I can run, lint, type-check, and reload the app quickly and consistently.**
- Audit current dev tooling: ESLint config (UI + backend), Prettier, `vue-tsc` strictness, backend `tsc` strict flags
- Document common workflows in `README.md` (where missing)
- Verify HMR works reliably in Docker (the `usePolling: true` workaround is fine but document why)
- Add minimal pre-commit checks (typecheck on staged files, optional)

### US-002 (P0) — Frontend performance pass
**As a user on a mid-range phone, the app loads in under 2s on a warm cache.**
- Measure: Lighthouse mobile score, bundle size (`yarn build` + `dist/` analysis)
- Identify wins: route-level code-splitting (already partial), lazy-load MapLibre on `/map` only, image lazy-loading on beach photos, font subsetting
- Set a perf budget in `vite.config.ts` build warnings
- Profile MapLibre cold-start on `/map` first visit

### US-003 (P0) — Backend performance & resilience
**As a user, the API stays responsive even while the forecast fetcher runs.**
- Measure baseline `/beaches` and `/beaches/:id/forecast` p50/p95 response times
- Verify cron-triggered forecast fetch doesn't block the HTTP server (move to worker if it does)
- Review MongoDB indexes on `beachId`, `fetchedAt`
- Add structured logging (request id, duration, status) — minimal, no new framework

## Theme 2 — Scoring engine maturity (P0–P1)

The score is the product. Two related capabilities: a way to **validate** it against reality, and the **admin tools** to act on what we learn.

### US-004 (P0) — Forecasting validation tool
**As the maintainer, I can record what the surf was actually like at a beach and compare it to what the score predicted.**
- Admin-only route (e.g. `/admin/validation`)
- Form: pick a beach + date/time → see predicted score + reasons → enter observed conditions (height, period, wind, "was it actually surfable: yes/no/somewhat") + free-text notes
- Stored in a new MongoDB collection `validationReports`
- List view: predicted vs. observed delta over time, per beach
- This is the data feedback loop that drives every later profile tweak

### US-005 (P1) — Admin "Fetch forecasts now" button
**As the maintainer, I can manually trigger a forecast refresh from the admin UI without hitting the API by hand.**
- Small admin-only page (or a control on the validation page from US-004) with a single "Fetch forecasts now" button
- Calls `POST /admin/fetch-forecasts` (already implemented backend-side)
- Shows in-flight state (spinner / disabled button) and a success/failure toast with summary (e.g. "8 beaches refreshed, 0 errors")
- Useful for: validating the new validation tool with fresh data, testing scoring tweaks immediately after a profile edit (US-007), or recovering after a missed cron run
- Establishes the admin-route gating pattern (auth, layout, toast UI) that US-006 / US-007 / US-017 will reuse

### US-006 (P1) — Beach profile editor (admin UI)
**As the maintainer, I can view and edit beach profiles in the browser instead of hand-editing JSON.**
- New Lit components for the form (sliders for ranges, compass for direction windows)
- Reads/writes profiles either to `beach_profiles.json` (via backend write) or to a `beachProfiles` MongoDB collection — **decision needed**
- Validate before save (direction window sanity, period/height ranges non-empty)
- Triggers re-score for the affected beach on save (can leverage US-005's fetch flow)

### US-007 (P1) — Beach addition workflow
**As the maintainer, I can add a new beach via the admin UI.**
- Builds on US-006's form
- Includes coordinate picker (drop a pin on a small MapLibre instance)
- Optional: profile-suggestion preview (see US-017) to seed defaults

### US-008 (P1) — Physical validation feedback loop
**As a user (and the maintainer), I can capture quick session notes that feed back into profile tuning.**
- After a session, mark a forecast hour as "I was there — it was [great / okay / not surfable]"
- Stored locally in IndexedDB initially; syncs to `validationReports` if backend is reachable
- Surfaces aggregated drift to the maintainer (e.g. "Mesachti consistently scores 0.6 when users report 'great'")

## Theme 3 — Forecast visualisation in beach detail (P1)

Surfers don't pick a day, they pick a window. Charts make that window visible.

### US-009 (P1) — Hourly forecast bar chart
**As a user, I can see hour-by-hour surfability across the next 10 days as a single chart.**
- X-axis: time (hourly, scrollable)
- Bars: surfability score, coloured by label (poor → very good)
- Optional overlay lines: swell height (m) and period (s) on a secondary axis
- Tap a bar → tooltip with full reasons for that hour

### US-010 (P1) — Radar chart for current conditions
**As a user, I can see at a glance which factors are good and which are limiting right now.**
- Axes: swell direction match, period match, height match, wind suitability
- Each axis 0–1; the polygon shape immediately shows "what's missing"
- Optional comparison overlay: "ideal" polygon (1.0 on every axis) vs. current

### US-011 (P1) — "Wave hunting" affordances
**As a user, when I tap a forecast hour, I see what the bottleneck is.**
- Tooltip says e.g. "Almost ideal — but wind is 25 km/h cross-shore"
- "Best window today" call-out at the top of beach detail (peak hour + score + duration)
- Filter chips on the chart: "above maybe", "above good" — dim everything below

### Open question — chart library
| Library | Bundle (gz) | Pros | Cons |
|---|---|---|---|
| **Chart.js** + `vue-chartjs` | ~60 KB | Native bar/line/radar; canvas-based; broad ecosystem | Tooltips/customisation can get fiddly |
| **Apache ECharts** | ~140 KB | Polished interactions; rich radar/heatmap support | Heaviest; API surface is large |
| **uPlot** | ~40 KB | Extremely fast for time-series | No radar; minimal styling |
| **D3** (custom) | ~70 KB (subset) | Total flexibility | Build-yourself overhead |

**Recommendation:** start with **Chart.js + vue-chartjs**. Native support for both bar (US-009) and radar (US-010), reasonable bundle size, escape hatch to ECharts later if interactions grow.

## Theme 4 — Map UX & coastline discovery (P1–P2)

Today's map shows everything; what we want is a map that shows **beaches first, roads to beaches second, and quietly hides the rest** — plus a few high-leverage scouting overlays so the map becomes a forecasting tool, not just a locator.

**Layer-priority note for Greece:** the Aegean is mostly windswell-driven and short-period, so wind-direction overlays carry more decision weight than swell-period overlays. The Ionian and the south coast of Crete pick up Mediterranean groundswell more reliably — weight layer prominence accordingly. (This is also why the existing wind-arrow overlay was the right first map data layer to ship in Phase 13.)

### US-012 (P1) — Simplified map style
**As a user, opening the map immediately draws my eye to the coastline and the beaches.**
- Replace the demo basemap with a custom MapLibre style (`style.json`) that:
  - Mutes inland landcover (greys/sands instead of greens)
  - Hides minor labels and irrelevant POI categories
  - Keeps water and coastline crisp and high-contrast
- Beaches remain the colour-graded primary markers (already implemented)

### US-013 (P1) — Highlight beaches and access roads
**As a user planning a trip, I can see the road that connects me to the beach.**
- Beaches: primary highlight tier (existing markers)
- Access roads: secondary highlight tier — render OSM `highway=primary|secondary|tertiary` segments within ~2 km of each configured beach in an accent colour
- Implementation options:
  - Pre-compute the access-road geometry per beach at seed time and serve as GeoJSON
  - Or: filter live from the basemap style's road layers using a coordinate-bbox expression
- All other roads dimmed to neutral grey

### US-014 (P1) — Bathymetry overlay (toggleable)
**As a user evaluating a beach, I can toggle a bathymetry layer to see the seafloor contours that shape waves.**
- Add a raster source: **EMODnet Bathymetry** WMS tiles (higher resolution, European waters — directly relevant to Greek coast); fall back to **GEBCO** WMS for global coverage if EMODnet is unavailable
- Render as a semi-transparent raster layer above the basemap (~50% `raster-opacity`); toggle on/off via a control alongside the wind-arrow toggle
- What surfers learn: shallow shelves vs. abrupt drop-offs, reef structure, sandbar shape — the seafloor is what shapes a wave
- Attribution requirement: surface "© EMODnet Bathymetry" / "GEBCO Compilation Group" in the map's attribution control

### US-015 (P1) — Satellite basemap toggle
**As a user scouting a coastline, I can switch from the vector map to satellite imagery to see real reef and sandbar shapes.**
- Two registered MapLibre styles: vector (the minimal style from US-012) and satellite (Esri World Imagery — free for non-commercial use; verify license at ship time)
- Small map control to toggle (icon button); use `map.setStyle()` to swap
- Re-add custom layers (beach markers, wind arrows, bathymetry from US-014) on the `style.load` event so overlays survive the swap
- Why it matters: satellite at low tide can reveal the actual reef or sandbar — invaluable for evaluating an unknown spot

### US-016 (P2) — Offline map tiles via PMTiles
**As a user on a coastal road trip with patchy signal, the map keeps working from locally cached tiles.**
- Build a Greek-coast PMTiles archive (single file, region-bounded) — vector tiles for the simplified style from US-012
- Register the `pmtiles://` protocol with MapLibre via the official `pmtiles` JS library
- Serve the archive from the app's static assets, or cache it via the service worker on first online visit
- Falls back to online tiles when reachable; surfaces a "using offline tiles" indicator when not
- Aligns with the project's offline-first value already applied to scores and forecasts

### US-017 (P2) — Tap a coastline → propose a beach profile
**As an explorer, I can tap any coastline on the map and get a starter profile to evaluate that spot.**
- Click on map (only if not on an existing beach) →
  - Reverse-geocode coordinates → place name + region
  - (Optional) fetch nearby info from external sources for context
  - Compute coastline normal at that point (orientation hint)
  - Use a kNN over existing profiles (geographic + orientation similarity) to seed defaults: ideal swell window, period/height ranges, wind preference
  - Open the US-006 profile editor pre-filled → user edits + saves → US-007 add-beach flow
- This is the "make surfsapp grow itself" feature.

### Sub-decisions for US-017
- **Geocoding source** — Nominatim (free, OSM ToS rate-limited) vs. Mapbox Geocoding (paid, robust, 100k/mo free tier). Lean Nominatim for v0.2 with caching.
- **External info source** — Wikipedia REST API for nearby articles, OSM Overpass for tagged surf-related POIs, or skip in v0.2 and add in v0.3.
- **Profile-suggestion algorithm** — start simple: nearest-neighbour by haversine distance + orientation difference; refine after we have validation data from US-008.
- **Auth gating** — same gating as US-006 admin UI.

## Cross-cutting open questions

1. **Admin-route auth** — for v0.2, is local-only / hidden route enough, or do we want a basic password gate? Simplest: env-var-based shared secret + middleware.
2. **Profile storage** (US-006) — keep `beach_profiles.json` as source of truth and make the admin UI a JSON editor + commit-on-save? Or migrate to a `beachProfiles` MongoDB collection? Trade-off: git history vs. live editing without redeploys.
3. **Map basemap hosting** — switch from MapLibre demo tiles to a self-hosted minimal style (PMTiles)? Affects US-012/US-013 implementation.
4. **Validation data ownership** (US-004 / US-008) — is the validation data per-user (local-only) or shared (server-side)? For v0.2 it's likely just the maintainer.

## Out of scope for v0.2 (deferred to v0.3+)

**Product:**
- User accounts and cloud sync of favourites
- Push notifications ("today is surfable at your favourite beach")
- Capacitor native packaging
- Multi-region expansion beyond Greece
- Tide integration
- Copernicus Marine API integration
- Community condition reports

**Map features evaluated and deferred** (revisit once v0.2 map work lands and we have validation data):
- **Time-animated marine weather raster overlay** (scrub swell height/period/direction over the next 7 days) — partially overlaps Theme 3 charts; large effort to build a timestamped tile stack + slider; revisit after charts ship
- **Bearing + pitch swell-orientation tool** (`map.setBearing/setPitch` + dynamic swell-direction arrow at a tapped point) — high conceptual value, but needs UX design pass; a candidate for v0.3 once US-017 (coastline-tap) lands
- **3D terrain rendering for headlands / wind shelter** — perf-heavy, niche use case (matters most for offshore-wind nuance between adjacent beaches); reconsider once we have a self-hosted DEM
- **Hillshade rendering of bathymetric DEM** — depends on terrain-RGB bathymetry tile availability (not a free public source); evaluate after US-014's bathymetry layer is in place
- **OSM Overpass `sport=surfing` import as supplementary GeoJSON layer** — sparse coverage in Greece (most spots are not tagged); curated profiles + US-017 coastline-tap give better signal
- **`maplibre-gl-draw` measurement / sketch tool** (paddle distance, rip channels, lineup notes) — power-user feature; evaluate after physical-validation feedback (US-008) tells us whether users want it
