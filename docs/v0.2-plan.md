## Phase 1 вЂ” Foundation (P0)

**Goal:** Lock dev tooling, baseline performance, and observability so later phases can be measured against a known starting point.

- **Stories:** US-001, US-002, US-003
- **Pre-requisite decisions:** none
- **Deliverables:**
  - ESLint + Prettier config in `ui/` and `backend/`; `yarn lint` / `npm run lint` scripts
  - Documented HMR/Docker workflow in `README.md`
  - Lighthouse mobile baseline report committed to `docs/perf-baseline.md`
  - Bundle-size baseline + perf budget in `vite.config.ts`
  - Backend p50/p95 latency baseline for `/beaches` and `/beaches/:id/forecast`
  - Structured request logging middleware (request id, duration, status)
  - MongoDB indexes verified on `forecastSnapshots.beachId` and `forecastSnapshots.fetchedAt`
- **Exit criteria:**
  - Lint passes clean across both packages
  - Lighthouse mobile score and bundle size recorded as the v0.2 baseline
  - Backend latency baseline recorded; cron fetch confirmed not to block API (or moved to worker if it does)

## Phase 2 вЂ” Validation loop + admin foundations (P0вЂ“P1)

**Goal:** Stand up the admin route + auth gating pattern and the validation data store so we can start measuring scoring accuracy.

- **Stories:** US-004 (P0), US-005 (P1)
- **Pre-requisite decisions:** D1 (admin auth), D7 (validation data ownership)
- **Deliverables:**
  - `/admin/*` routes gated by env-var shared-secret middleware
  - New MongoDB collection `validationReports` with Mongoose schema
  - `/admin/validation` page (Vue view) with predicted-vs-observed form + list
  - "Fetch forecasts now" button calling existing `POST /admin/fetch-forecasts`, with toast UX
  - Reusable Lit admin shell (header, layout, toast component) consumed by US-005 and later admin views
- **Exit criteria:**
  - Validation reports can be created, listed, and queried per beach
  - Manual fetch trigger works end-to-end and surfaces success/error counts
  - Admin gating denies unauthenticated access; secret rotation works via env

---

## Phase 3 вЂ” Profile tooling (P1)

**Goal:** Stop hand-editing `beach_profiles.json`; enable adding new beaches via UI.

- **Stories:** US-006, US-007
- **Pre-requisite decisions:** D2 (profile storage)
- **Deliverables:**
  - Beach profile editor page under `/admin/profiles`
  - Lit components: range slider, direction-window compass, profile form
  - Coordinate-picker (mini MapLibre instance) for new-beach flow
  - Save handler that persists per D2 and triggers a re-score (reuses US-005 fetch path)
- **Exit criteria:**
  - Editing a profile + saving updates `beach_profiles.json` (or DB) and re-scores within one cycle
  - New beach can be added end-to-end from UI and appears on `/beaches`
  - Validation prevents saving illegal direction windows or empty ranges

## Phase 4 вЂ” Forecast visualization (P1)

**Goal:** Make hourly conditions and limiting factors visible in beach detail. **Independent of Phases 2вЂ“3 вЂ” can run in parallel.**

- **Stories:** US-009, US-010, US-011
- **Pre-requisite decisions:** D3 (chart library)
- **Deliverables:**
  - Chart.js + `vue-chartjs` added to `ui/package.json`
  - Hourly bar chart component (10-day, scrollable, label-coloured)
  - Radar chart component (4 axes: swell-dir match, period match, height match, wind suitability)
  - "Best window today" call-out + bottleneck tooltip text in beach detail
  - Filter chips on the bar chart ("above maybe", "above good")
- **Exit criteria:**
  - Beach detail renders both charts on data from `/beaches/:id/forecast`
  - Tap on any hour shows reasons including the limiting factor
  - Bundle increase from charts measured against Phase 1 budget; within tolerance

## Phase 5 вЂ” Map scouting upgrade (P1)

**Goal:** Turn the map from locator into forecasting tool. **Independent of Phases 2вЂ“3 вЂ” can run in parallel.**

- **Stories:** US-012, US-013, US-014, US-015
- **Pre-requisite decisions:** D4 (bathymetry source), D5 (satellite licensing)
- **Deliverables:**
  - Custom MapLibre `style.json` (minimal vector basemap)
  - Access-road GeoJSON layer (pre-computed at seed time for the curated beaches)
  - Bathymetry raster source (EMODnet primary, GEBCO fallback) + opacity slider/toggle
  - Satellite style (`style-satellite.json`) + style-toggle control
  - Layer-restoration logic on `style.load` to preserve markers/wind arrows/bathymetry across `setStyle()` swaps
  - Attribution control surfaces all layer credits
- **Exit criteria:**
  - Map opens to the simplified vector style by default
  - Toggling bathymetry, satellite, and wind-arrow each works without losing other layers
  - Access-road layer renders only the configured beaches' connecting roads in the accent colour
  - All third-party data sources display attribution per their licenses

## Phase 6 вЂ” User feedback loop (P1)

**Goal:** Close the loop from end users back to validation data so profile drift surfaces automatically.

- **Stories:** US-008
- **Pre-requisite decisions:** none beyond Phase 2
- **Deliverables:**
  - Per-hour "I was there" tagging UI in beach detail
  - Local IndexedDB store for offline session reports (Dexie schema addition)
  - Sync handler that posts cached reports to `validationReports` when online
  - Aggregated drift indicator on the admin `/admin/validation` view
- **Exit criteria:**
  - User can tag a forecast hour offline; the report syncs on next online visit
  - Drift indicator appears for any beach with в‰Ґ3 reports
  - Reports are tied to the same `validationReports` collection used by US-004

## Phase 7 вЂ” Stretch (P2)

**Goal:** Explorer-grade features. Optional for v0.2.

- **Stories:** US-016 (offline PMTiles), US-017 (coastline tap в†’ profile)
- **Pre-requisite decisions:** D6 (PMTiles size budget); US-017 also needs Phase 3 to be complete
- **Deliverables:**
  - Greek-coast PMTiles archive + `pmtiles://` protocol registration
  - Service-worker caching of the archive on first online visit
  - Reverse-geocoding integration (Nominatim, with caching)

  - kNN profile-suggestion algorithm (haversine + orientation similarity over existing profiles)
  - Tap-coast handler that pre-fills the US-006 editor
- **Exit criteria:**
  - Map remains interactive offline (zoom/pan) with cached tiles
  - Tapping unconfigured coast opens the profile editor with sane defaults
  - End-to-end: tap coast в†’ review/adjust profile в†’ save в†’ beach appears on `/beaches`

## Risk register

| Risk | Phase | Mitigation |
|---|---|---|
| Cron-triggered forecast fetch blocks the API under load | 1 | Measure in Phase 1; move fetcher to a worker process if p95 regresses during a fetch cycle |
| MapLibre `setStyle()` drops custom layers (markers, wind arrows, bathymetry) | 5 | Centralise layer-add logic in a `applyOverlays()` function; call on every `style.load` |
| EMODnet WMS rate-limits or has downtime | 5 | Smoke-test in D4; ship GEBCO as runtime fallback if EMODnet errors |
| PMTiles archive too large to cache via service worker | 7 | Trim zoom range (4вЂ“13) and bbox to Greek coast only; budget < 100 MB; defer to v0.3 if still too large |
| Profile JSON edits via admin UI lose git history | 3 | Per D2, commit-on-save (server runs `git commit` on the seed file) вЂ” or accept and migrate to DB collection in v0.3 |
| Charts add unacceptable bundle weight | 4 | Phase 1 sets the budget; lazy-load Chart.js only on beach detail route |

## Definition of done (per story, applied at phase exit)

Every story must satisfy all of the below before its phase is considered complete:

- `yarn typecheck` (UI) and `npm run typecheck` (backend) pass clean
- `yarn lint` / `npm run lint` pass clean (once Phase 1's lint config is in)
- Manual smoke check on real iOS Safari and Android Chrome (via the cloudflared tunnel workflow documented in `README.md`)
- No regression vs Phase 1 perf baseline (Lighthouse mobile score, bundle size, backend p95)
- Code reviewed against the brief's acceptance bullets for that story
