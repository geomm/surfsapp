## Codebase Patterns
- Backend uses `tsx` for running TypeScript directly (no compile step needed for scripts)
- Backend typecheck: `cd backend && npm run typecheck`
- Beach profile JSON data lives in `backend/src/data/`; docs/beach-profiles.md is the source of truth for profile values
- module: "commonjs" in tsconfig — no .js extensions needed in imports
- idealWindDescription union values: 'wave-generating-onshore' | 'offshore-or-light'
- skillLevel union values: 'beginner' | 'intermediate' | 'advanced'
- label/overallLabel union values: 'poor' | 'maybe' | 'good' | 'very-good'
- Sub-document schemas (HourlyForecast, DailySummary) use `_id: false` to avoid unnecessary _id generation
- Use `.lean().select('-_id -__v')` to strip MongoDB internals from responses
- backend/src/db/connection.ts exports connectDB() — needed by any script that talks to MongoDB
- Open-Meteo time strings lack a Z suffix — append 'Z' when constructing Date objects so they parse as UTC
- Use `beach.offshoreCoords ?? beach.coords` for Open-Meteo API coords (offshore preferred)
- forecastFetcher.ts lives at backend/src/services/forecastFetcher.ts — contains fetchForecastForBeach, saveForecastSnapshot, fetchAllBeaches
- ForecastSnapshot model at backend/src/models/ForecastSnapshot.ts — IHourlyForecast, IDailySummary, IForecastSnapshot interfaces
- Beach model at backend/src/models/Beach.ts — IBeach interface, BeachSchema, Beach model
- windScoringLogic field uses lowercase 'p' in messinesspenalty — match exactly, do NOT rename to camelCase
- rawData keys match Open-Meteo variable names: swell_wave_height, swell_wave_period, swell_wave_direction, swell_wave_height_2, swell_wave_direction_2, wind_speed_10m, wind_direction_10m, wave_height
- Direction ranges crossing 360° appear in beach profiles (e.g. [340, 20]) — all angular comparisons must handle wraparound
- Scoring algorithm details in docs/scoring-engine.md — hard gates, subscores, weighted sum, label mapping, daily summaries
- scoringEngine.ts lives at backend/src/services/scoringEngine.ts — exports angularDistance, isAngleInRange, applyHardGates (and later subscore/scoreHour functions)
- Angular distance formula: `Math.abs(((a - b + 540) % 360) - 180)` — returns 0–180, handles wraparound
- Compass range midpoint: `span = ((to - from) % 360 + 360) % 360`, `midpoint = (from + span/2) % 360` — handles 360° crossing
- Subscore functions follow continuous gradient pattern with `lerp()` helper, not discrete buckets
- scoringEngine.ts exports: angularDistance, isAngleInRange, applyHardGates, scoreSwellDirection, scoreSwellPeriod, scoreSwellHeight, scoreWind, scoreHour, computeDailySummaries
- Wind scoring uses dual-mode: if beach has windScoringLogic AND wind within ±30° of generating wind dir → WGO logic, else → standard logic
- Standard wind scoring: windAngle = angularDistance(windDir, shorelineNormalDeg), then bands for offshore/cross/onshore
- WGO wind scoring: additive modifiers from baseline 50 (+20 generating, +25 quality, -35/-21 messiness)
- scoreHour returns `label: string` — cast to `IHourlyForecast['label']` when assigning to IHourlyForecast objects
- forecastFetcher.ts imports scoreHour and computeDailySummaries from scoringEngine.ts — scoring is wired into the fetch pipeline

## 2026-04-06 - US-003
- Implemented three swell subscore functions in `backend/src/services/scoringEngine.ts`:
  - `scoreSwellDirection(beach, swellDir)` — continuous gradient from ideal centre (100) → ideal edge (80) → acceptable edge (40) → 0 outside
  - `scoreSwellPeriod(beach, period)` — linear interpolation across min (0–20), sub-ideal (20–60), ideal (70–100), above-ideal (80–60) ranges
  - `scoreSwellHeight(beach, height)` — linear interpolation with skill-level-dependent penalty above ideal range
- Added helper functions: `rangeMidpoint()` for compass direction midpoint with 360° wraparound, `lerp()` for linear interpolation
- Files changed: `backend/src/services/scoringEngine.ts`
- **Learnings for future iterations:**
  - Compass range midpoint formula: span = ((to - from) % 360 + 360) % 360, midpoint = (from + span/2) % 360
  - For direction scoring, acceptable half-width should use max of distances from ideal midpoint to both acceptable edges (handles asymmetric acceptable ranges)
  - Period scoring inside ideal range uses symmetric triangle: 70 at edges, 100 at midpoint — same pattern for height (80 at edges, 100 at midpoint)
  - Above-ideal period decays towards idealHigh*2 as the far end (80→60)
  - Above-ideal height penalty uses overFactor = (height - idealHigh) / idealHigh, clamped to [0,1]
